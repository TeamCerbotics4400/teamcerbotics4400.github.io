1. Chasis (DriveTrain)

This class controls the robot's tank-like chassis, which consists of two motors (one right and one left).

The constructor retrieves references to the motors from the HardwareMap, configures the braking

behavior (BRAKE) so the robot stops abruptly instead of skidding when power is lost, and reverses the

direction of the left motor so both motors rotate coherently when receiving the same signal. The drive()

method combines the forward (speed) and turn (turn) values ​to calculate the power for each motor,

allowing the robot to move forward, backward, and in combined turns.

package org.firstinspires.ftc.teamcode;

import com.qualcomm.robotcore.hardware.DcMotorEx;

import com.qualcomm.robotcore.hardware.DcMotorSimple;

import com.qualcomm.robotcore.hardware.HardwareMap;

public class DriveTrain {

DcMotorEx rightMotor;

DcMotorEx leftMotor;

HardwareMap hw;

public DriveTrain(HardwareMap hw){

this.hw = hw;

rightMotor = hw.get(DcMotorEx.class, "rightMotor");

leftMotor = hw.get(DcMotorEx.class, "leftMotor");

rightMotor.setZeroPowerBehavior(DcMotorEx.ZeroPowerBehavior.BRAKE);

leftMotor.setZeroPowerBehavior(DcMotorEx.ZeroPowerBehavior.BRAKE);

leftMotor.setDirection(DcMotorSimple.Direction.REVERSE);

}

public void drive(double speed, double turn){

double rightPower = speed - turn;

double leftPower = speed + turn;

rightMotor.setPower(rightPower);

leftMotor.setPower(leftPower);

}

}

2. Subsystems (Subsystems)

This class groups the robot's mechanisms that are not part of the chassis: the climber motor and the

intake motor. In the constructor, both motors are obtained from the HardwareMap, and the climber's

automatic brake is activated to maintain its position when it is not receiving power. The

`setPowerClimber()` and `setPowerIntake()` methods allow the power of each motor to be controlled

independently from the TeleOp.

package org.firstinspires.ftc.teamcode;

import com.qualcomm.robotcore.hardware.DcMotorEx;

import com.qualcomm.robotcore.hardware.HardwareMap;

public class Subsystems {

DcMotorEx climber;

DcMotorEx intake;

HardwareMap hw;

public Subsystems(HardwareMap hw){

this.hw = hw;

climber = hw.get(DcMotorEx.class, "climber");

climber.setZeroPowerBehavior(DcMotorEx.ZeroPowerBehavior.BRAKE);

intake = hw.get(DcMotorEx.class, "intake");

}

public void setPowerClimber(double power){

climber.setPower(power);

}

public void setPowerIntake(double power){

intake.setPower(power);

}

3.5 Control del intake

With the state of the bumpers already read, this block decides the power of the intake: if the left bumper

is pressed, the intake rotates inwards (power 1); if the right one is pressed, it rotates outwards (power -1);

and if neither is pressed, the intake stops (power 0).

if(isLeftBumpPressed){

m_subsystems.setPowerIntake(1);

} else if(isRightBumpPressed){

m_subsystems.setPowerIntake(-1);

}

else{

m_subsystems.setPowerIntake(0);

}

3.6 Control del escalador

Similarly, this block uses the D-pad to control the climber: D-pad up raises it (power 1), D-pad down

lowers it (power -1), and if neither is pressed, the motor stops (power 0). Since the climber's motor is set

to automatic braking (BRAKE), at power 0 it will maintain its position

instead of falling freely.

4.0 Blocks

This FTC code project is organized into three parts: the DriveTrain class controls the robot's tank-style

chassis, made up of two motors (right and left), where the constructor gets the motor references from the

HardwareMap, sets automatic braking (BRAKE) so the robot stops sharply instead of coasting, and

reverses the direction of the left motor so both motors spin consistently, while the drive() method

combines the forward value (speed) and the turn value (turn) to calculate the power for each motor,

moving the robot forward, backward, and through combined turns; the Subsystems class groups the

robot's additional mechanisms — the climber motor and the intake motor — getting both from the

HardwareMap in the constructor and enabling automatic braking on the climber so it holds its position

when it isn't receiving power, while the setPowerClimber() and setPowerIntake() methods allow each

motor's power to be controlled independently; and finally the MyTeleOp class, which extends

LinearOpMode and is the program that runs during the match's manual-control period, declares the

m_drive and m_subsystems variables to hold the chassis and subsystems instances, initializes them with

the hardwareMap when the OpMode starts, sends a status telemetry message, and waits with

waitForStart() for the operator to press PLAY, and once the match begins enters a loop that, while the

OpMode is active, reads gamepad 1's sticks to drive the chassis, reads the D-Pad and bumpers to know

which buttons are pressed, controls the intake so it spins inward with the left bumper, outward with the

right bumper, and stops if neither is pressed, and controls the climber similarly using the D-Pad to raise it,

lower it, or hold it in place thanks to the automatic brake configured on its motor.
